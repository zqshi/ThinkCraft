# 设计原则检查清单

> 确保产品设计符合核心设计原则

---

## 用户中心原则

### 场景真实性

- [ ] 场景描述具体真实，非虚构或假设
- [ ] 场景来源于真实用户调研或数据分析
- [ ] 场景包含具体的触发条件和上下文

### 价值导向

- [ ] 明确说明为用户创造的具体价值
- [ ] 价值可量化或可验证
- [ ] 价值与用户痛点直接相关

### 渐进式

- [ ] 功能设计支持渐进式学习
- [ ] 新用户能快速上手核心功能
- [ ] 高级功能不干扰基础使用

---

## 数据驱动原则

### 证据基础

- [ ] 设计决策基于数据和事实
- [ ] 引用的数据来源可靠且已标注
- [ ] 避免主观臆断和无根据的假设

### 可验证性

- [ ] 设计假设可通过数据验证
- [ ] 定义了明确的验证指标
- [ ] 验证方法可执行（如 A/B 测试、用户调研）

### 迭代优化

- [ ] 设计支持快速迭代
- [ ] MVP 定义清晰，聚焦核心价值
- [ ] 有明确的迭代优化计划

---

## 系统性原则

### 完整性

- [ ] 功能设计覆盖完整的用户旅程
- [ ] 考虑了正常、异常、边界情况
- [ ] 没有明显的功能缺失或逻辑断层

### 一致性

- [ ] 术语使用一致
- [ ] 交互模式一致
- [ ] 视觉风格一致（如适用）

### 扩展性

- [ ] 设计支持未来功能扩展
- [ ] 没有硬编码限制
- [ ] 接口设计灵活可扩展

---

## 边界原则

### 排除技术实现细节

- [ ] 没有涉及技术架构设计
- [ ] 没有涉及技术栈选择
- [ ] 没有涉及 API 设计
- [ ] 没有涉及数据库 Schema
- [ ] 没有涉及部署方案

### 聚焦产品层面

- [ ] 聚焦用户需求和业务价值
- [ ] 聚焦功能设计和交互逻辑
- [ ] 聚焦用户故事和验收标准

---

## 大模型原生应用原则

### 功能优先（而非流程优先）

- [ ] 设计聚焦功能点，而非固定流程
- [ ] 用户可在任意节点触发核心功能
- [ ] 没有强制的线性步骤引导

### 任意节点触发

- [ ] 核心功能可独立触发
- [ ] 不依赖全局流程状态
- [ ] 前置条件仅限该功能必要数据

### 无全局流程依赖

- [ ] 各功能自洽独立
- [ ] 没有跨功能的强制顺序耦合
- [ ] 功能间通过数据依赖而非流程依赖

### 可测试性

- [ ] 验收标准覆盖正常/异常/边界场景
- [ ] 使用 Given-When-Then 格式
- [ ] 验收标准可直接转化为测试用例

### 结构化与可解析

- [ ] 输出采用结构化格式
- [ ] 功能描述/业务规则/输入输出/状态管理清晰
- [ ] 机器可解析，无歧义

### 约束显式

- [ ] 隐私、合规、禁止性规则显式列出
- [ ] 不隐含在叙述中
- [ ] 约束条件明确可验证

---

## 反模式检查（禁止）

### 流程中心化

- [ ] 是否以流程为主导、功能附属？
- [ ] 是否强制用户按固定顺序操作？

### 跨功能强前置依赖

- [ ] 是否存在不必要的功能间顺序耦合？
- [ ] 是否可以解耦为独立功能？

### 不当度量

- [ ] 成功指标是否混入了与当前功能无直接依赖的业务 KPI？
- [ ] 成功指标是否可直接验证功能完成度？

---

## 审查清单（抽查要点）

- [ ] 是否功能为中心而非流程为中心？
- [ ] 是否可在任意节点触发核心功能且无全局流程依赖？
- [ ] 验收标准是否覆盖正常/异常/边界并可直接测试？
- [ ] 是否使用结构化格式且语言无歧义？
- [ ] 是否显式列出约束？
- [ ] 是否存在反模式迹象（流程中心化/强耦合/不当度量）？

---

## 使用说明

### 何时使用？

- 需求设计文档完成后
- 战略设计文档完成后
- 质量挑战阶段
- 任何需要验证设计质量的时候

### 如何使用？

1. 逐项检查清单
2. 标记未通过的项
3. 针对未通过项提出改进建议
4. 重新设计后再次检查

### 谁来使用？

- product-demand-manager-agent（自检）
- product-demand-challenge-agent（质量挑战）
- strategy-design-agent（自检）
- strategy-design-challenge-agent（质量挑战）
- 人类审查者（最终验收）
