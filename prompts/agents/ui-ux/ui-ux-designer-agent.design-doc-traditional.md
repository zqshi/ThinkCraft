- 完成项目研究分析后
- 需要产出正式需求文档时
- 需要与团队和利益相关者沟通需求时

5. **可测试性**: 验收标准必须可测试

1. **结构化**: 使用JSON格式，机器可解析
2. **精炼**: 去除冗余信息，保留核心内容
3. **明确**: 字段定义明确，无歧义
4. **完整**: 包含所有必要信息
5. **可测试**: 验收标准可直接转化为测试用例

### 与传统版的区别
| 维度 | 传统版 | LLM版 |
|------|--------|-------|
| 目标读者 | 人类 | LLM |
| 格式 | Markdown + 表格 | JSON |
| 详细程度 | 详细，包含背景和说明 | 精炼，只保留核心信息 |
| 可读性 | 高（人类） | 高（机器） |
| 可解析性 | 低 | 高 |

1. 将精炼需求文档传递给战略设计Agent

## 使用说明

### 何时使用？

- 收到需求设计挑战文档后
- 需要回应挑战问题时
- 需要人类裁决时

### 回应原则

1. **客观理性**: 回应要客观理性，基于事实和原则
2. **具体明确**: 回应要具体，避免模糊表述
3. **建设性**: 即使不接受，也要说明理由并提出替代方案
4. **保持专业**: 保持专业态度，避免情绪化

### 人类裁决

- **何时需要**: 挑战方和回应方无法达成一致时
- **裁决结果**: 支持挑战方/支持回应方/折中方案

### 后续流程

1. 根据人类裁决结果修改需求设计文档
2. 产出"精炼需求文档-LLM版"
3. 进入战略设计阶段
